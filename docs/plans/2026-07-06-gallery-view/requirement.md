## 背景

当前项目已经从"单memo单兵作战"的笔记工具，演进为类似语雀的知识库产品（详见 [hierarchical-notes 需求](../2026-07-03-hierarchical-notes/requirement.md)），文档以「知识库 + 层级路径」组织。在此基础上，项目已额外支持 `html`、`pdf` 两种"仅渲染"文档类型（详见 [media_pdf 需求](../2026-07-04-media_pdf/definition.md)），可以在首页预览、explore 列表、doc 详情页等多处正确渲染。

本需求在此基础上，新增第三种特殊文档类型 —— **`view` 文档**，用于在某个知识库目录/标签节点下，以类似 Notion Gallery View 的相册卡片形式，组织展示该节点下的子文档。

## 术语

- **知识库（项目）**：hierarchical-notes 需求中定义的顶层容器，文档按目录路径归属其下。
- **view 文档**：本需求新增的第三种特殊文档类型（与 `html`、`pdf` 并列），其 content 只承载一份结构化配置（view 类型 + 展示范围 + 排序/分组/封面规则等），不承载渲染产物。
- **scope（展示范围）**：view 文档所配置的"要展示哪些子文档"，例如"当前目录下的直接子文档"或"某个 tag 下的所有文档"。
- **配置驱动渲染**：view 文档的 content 只是数据，实际的相册卡片是前端在打开该文档时，实时查询 scope 内的当前数据、用固定的 Gallery 组件渲染出来的——不做离线生成、不持久化渲染结果。

## 需求概述

1. 用户可以在知识库任意目录下新建一个 `view` 类型文档（类比新建一篇笔记/上传一个 html 文档的入口）。
2. 编辑该文档时，走引导式表单流程：选择 view 样式（当前只做 `gallery`）→ 出现该样式对应的配置表单（scope、排序方式、封面取值规则等）→ 表单顶部可选填一段 markdown 简介文本 → 提交后仅将"简介 markdown + 配置 JSON"存入该文档的 content，不写入任何 HTML。
3. 预览该文档时：
   - 若填写了简介，用**现有 markdown 渲染器**在相册视图上方原样渲染。
   - 相册主体部分，前端根据 content 中的配置，实时查询 scope 内的当前文档列表（标题、封面图、更新时间等属性），用固定的 Gallery 组件渲染成卡片墙。
4. `view` 文档不出现在 Explore 的 feed 列表中（它是结构化的组织节点，不是内容型笔记），但可以作为普通文档一样被目录树直接打开、被链接跳转到。

## 明确不做的事（对齐历史讨论结论，避免范围蔓延）

以下方案在设计讨论阶段被明确否决，写在这里防止后续实现时反复横跳：

1. **不引入独立的 `index.xxx` 物理文件概念。** 项目没有真实文件系统语义，view 的归属关系直接挂在知识库现有的目录/文档层级上，不额外造一层"目录配置文件"。
2. **不做"提交时生成最终 HTML 并持久化"的方案。** 会引入数据陈旧（子文档变化后相册不同步）、缓存失效管理、存储型 XSS 风险三个问题，且相较"实时查询+组件渲染"没有任何收益。content 里只允许出现"简介 markdown 文本 + 配置 JSON"，禁止出现渲染产物或用户手写的任意 HTML 片段。
3. **不做自由排布的多 block 编辑器**（即"在任意位置插入任意数量 md/html 块，可拖拽排序"的 Notion-block-editor 方案）。该方案的复杂度增量主要来自"块级编辑交互 + 有序数组数据模型"，而非渲染层面，属于另一个独立量级的功能（通用页面搭建器），不在本需求范围内。
4. **不引入 Formily 或任何 schema 驱动表单引擎。** 项目当前 view 类型数量少（仅 gallery，calendar 等留待后续），配置表单字段是工程师预先设计好的固定集合，不需要运行时可变的 schema 引擎；且项目现有前端栈是 Radix + 手写受控组件，没有 antd 等 Formily 生态适配的基建，引入成本大于收益。若未来 view 类型膨胀到 5 种以上，可重新评估，届时优先考虑 react-hook-form + zod（与现有栈更贴合），而非 Formily。
5. **简介字段是固定单个、位置写死在相册上方的 md 文本框，不做多槽位、不做自由定位。** 这是三档复杂度方案（固定单字段 / 固定多槽位 / 自由 block 数组）中评估后选定的最低复杂度档位，满足"能加点说明文字"的诉求即可，不追加更多灵活性。

## 数据模型

`view` 文档与 `html`、`pdf` 文档一样，作为知识库文档树中的一个节点存在，通过既有的 doc type 字段区分类型（复用 html/pdf 已经落地的"按类型分发渲染器"机制，不新增平行的判型逻辑）。

content 是一份结构化 JSON，概念上包含：

- `viewType`：当前固定为 `"gallery"`（预留未来 `"calendar"` 等扩展）。
- `description`：可选，纯 markdown 文本，展示在相册视图上方，复用现有 markdown 渲染管线（含既有的 `rehype-sanitize` 净化），不需要新的安全评估面。
- `scope`：展示范围描述（例如"当前目录直接子节点"或"指定 tag 下的全部文档"），具体取值规则由技术方案阶段结合现有目录/tag 数据模型敲定。
- 排序、分组、封面取值等 gallery 专属渲染参数。

content 中**不允许**出现任何 HTML 字符串、不允许出现渲染结果缓存字段。

## 编辑态流程

1. 用户新建 `view` 文档（或打开一篇已有的 `view` 文档进入编辑态）。
2. 若尚未选择 view 样式：展示样式选择入口（当前仅 gallery 一个选项，UI 上按可扩展的列表形式呈现，为后续 calendar 等预留位置）。
3. 选定样式后，展示该样式对应的**预置表单组件**（写死的 React 组件，非 schema 动态生成），字段包括：
   - 简介 markdown 文本框（可选，位置固定在表单顶部）
   - scope 选择器
   - 排序/分组/封面规则等 gallery 专属字段
4. 提交后，表单数据序列化为 content 描述的 JSON 结构，走既有的文档保存接口持久化。
5. 编辑态与预览态共享同一份 content：重新进入编辑态时，表单根据已存的 JSON 回填。

## 预览态流程

1. 打开 `view` 文档，前端识别其 doc type 为 `view`，分发给专属的 GalleryViewRenderer（与 html/pdf 的分发方式一致）。
2. 若 content 中 `description` 非空，用现有 markdown 渲染器原样渲染在相册区域上方。
3. 根据 content 中的 `scope`、排序/分组/封面规则，实时查询知识库当前数据（不读取任何历史缓存），渲染成 Gallery 卡片墙。
4. 数据源永远是查询时刻的最新状态：子文档标题变更、封面替换、新增/删除子文档，下次打开该 view 文档时自动反映，不需要任何"重新生成"操作。

## Explore 列表过滤

`view` 文档不应出现在 Explore 的 feed 流中（它是组织节点，不是可浏览的内容型笔记）。需要在 Explore 的 feed 查询逻辑中，按 doc type 排除 `view` 类型文档。此过滤规则应与 html/pdf 是否展示在 Explore 的既有规则放在同一处判断逻辑里维护，避免出现两套并行的"哪些 doc type 该出现在 feed 里"的判断代码。

（本条目在设计讨论阶段被标记为"后期实现"，此处正式收录为本需求的验收项之一，实现时不可遗漏。）

## 与已有 html/pdf 文档类型的关系

- `view` 是第三种"仅承载结构化数据、非自由文本"的特殊文档类型，复用 html/pdf 已经验证过的"按 doc type 分发渲染器"这套机制，不新增平行的类型判断体系。
- 若未来确有"在相册页面嵌入一段自定义静态 HTML"的诉求，应通过引用/嵌入已有的 `html` doc type 文档来实现，不在 `view` 文档 content 内部重新实现一套 HTML 承载能力（避免功能重复建设，详见"明确不做的事"第 2、3 条）。

## 后续可迭代方向（不在本次范围内，仅记录以防遗忘）

- 新增 `calendar` 等更多 viewType，复用 doc type 分发 + 预置表单的既有骨架。
- 若简介"固定单字段"在实际使用中体验不足，可评估升级到"固定多槽位"档位（例如相册上方 + 下方各一个 md 槽位），仍不引入自由 block 编辑器。
- 若 view 类型数量显著增加，重新评估表单方案是否需要从"预置组件硬编码"升级为"react-hook-form + zod"的半结构化方案。
